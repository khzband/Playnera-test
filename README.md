# Архитектура сцены
В качестве архитектурного паттерна этой сцены я выбрал MVP в разновидности Supervising Controller.

View. Ввод пользователя реализован через интерактивные элементы итерфейса. Сигналы с них поступают на Presenter.

Presenter. Этот блок представлен классом с логикой, которая обрабатывает сигналы с View и вызывает методы модели, которые изменяют ее состояние.

Model представлена двумя классами:
 - класс Model хранит в себе состояние куклы: цвета зон макияжа, состояние акне и необходимые логические флаги, с ними связанные.
 - класс UIView хранит состояние UI: выбранный инструмент, выбранный цвет, фаза цикла нанесения макияжа, состояние блокировок UI и логические флаги.

Особенность этой разновидности MVP в том, что между Model и View есть связь. Model может вызывать события об изменении своего состояния, а View эти события "слушает" и может считывать необходимые поля в Model для отображения изменений.

В такой реализации код получается более лаконичным и Presenter не распухает от того, что ему надо передавать сигналы в обе стороны. Model и View связаны слабо. View никак не может изменить состояние Model, т.к. у него нет методов, которые могут это сделать, а Model ничего не знает про View, только вызывает события.

# Базовые паттерны
Для установления связи между классами, я использовал Service Locator. По статической ссылке ServiceLocator.Instance любой класс может вызвать метод Get\<T>, чтобы получить доступ к зарегистрированному в этом сервисе классу. Service Locator Loader выполняет роль "точки входа": регистрирует необходимые классы и инициализирует их в нужной последовательности.

Для работы с событиями, реализован Event Bus в своем простейшем исполнении. Это класс, который с помощью делегатов Action, описывает все события, необходимые в сцене.

# Абстракция, полиморфизм и интерфейс IInstrument
Сцена предполагает сущности инструментов нанесения макияжа: кисть для румян, помада, кисть для теней и крем. У них явно есть два общих элемента поведения: переход в активированное состояние и использование с возвращением в начальное состояние. На этой основе я сделал интерфейс IInstrument, который реализован во всех классах инструментов и объявляет два метода: GetReady() и ApplyInstrument().

Их реализация у конкретных инструментов разная, но это позволяет нам сделать единый обработчик InstrumentController, который хранит ссылки на все инструменты, "подает" нужный инструмент и запускает процесс использования.

# Addressables
Поскольку в сцене потенциально может быть задействовано большое количество спрайтов большого размера, есть необходимость загружать их динамически и освобождать память, после их деактивации. Для этого использована система Addressables.

# Перемещения инструментов
Интересной особенностью сцены является то, что нам нужны как методы анимации перемещения инструментов с заданной скоростью, так и методы с заданным временем перемещения.

Методы с постоянной скоростью нужны для анимации перемещения между контрольными точками. Так мы гарантируем то, что скорость перемещения не будет зависеть от размера экрана устройства.

Методы с заданным временем нужны для анимации нанесения цвета на инструменты и нанесения макияжа. Это нужно для синхронизации нескольких систем, которые одновременно работают в этот момент. Например, при нанесении теней на глаза, за время перемещений кисти должен стать полностью непрозрачным спрайт с макияжем глаз. А также в реальном проекте будет звук нанесения макияжа, который должен быть синхронизирован с перемещениями кисти.

# Плавные переходы
Для плавного изменения зон макияжа, я использовал по два изображения на каждую зону. Они по очереди динамически загружают спрайты и при смене цвета макияжа старый цвет плавно становится прозрачным, а новый полностью непрозрачным.

Так мы получает красивый плавный переход одного цвета в другой.
